### The Problem

SEO216 needed a website that does more than look good — it had to rank on Google, get cited by AI assistants (ChatGPT, Perplexity, AI Overviews), and convert visitors into leads. The site operates in two languages (DE/EN) for the DACH market, requiring a locale architecture that serves native German slugs at the root while maintaining full English parity under `/en`.

### Solution

A maximized SEO / GEO / SXO website built on Astro + Vercel — statically prerendered, bilingual, with a machine-readable AI layer that goes beyond what most agencies build. Every page emits structured data as a connected entity graph, Markdown twins for AI crawlers, and a curated `llms.txt` for answer engines.

### Key Features

#### 1. Locale Architecture with Native Slugs

German lives at the root (`/leistungen`), English under `/en/services`. A `DE_SLUG_OVERRIDES` map handles native slug translation — prefix-matched so child paths like `/services/build` automatically become `/leistungen/build`. The inverse `delocalizePath()` powers hreflang alternate computation. TypeScript enforces a parity contract: every page exists fully in both locales or declares itself single-locale.

#### 2. Entity Graph Structured Data

One `@graph` per page with stable `@id`s — `Organization`, `WebSite`, `WebPage`, `BreadcrumbList`. Each page type adds its own schema: `Service` + `FAQPage` on service pages, `BlogPosting` on blog posts, `DefinedTermSet` on the glossary hub, `SoftwareApplication` on the Spindle page. Every schema node uses absolute URLs and references the same Organization entity, so Google sees a connected knowledge graph instead of isolated blobs.

#### 3. GEO Readability Layer

Three components work together to make the site citable by AI answer engines:

- **`robots.txt`** per-vendor allowlist — GPTBot, OAI-SearchBot, ClaudeBot, PerplexityBot, Google-Extended, and others are explicitly allowed. GoogleOther (training-only) is denied from `/llms.txt` so curated context isn't used as training material.
- **`llms.txt`** — curated company description, services, target industries, process, and links to `.md` fact sheets. One hour of work, zero cost, the fastest single step toward GEO visibility.
- **`.md` twin pages** — every marketing page has a Markdown version at `/<path>.md` (e.g. `/leistungen.md`). Blog posts re-emit their full body as Markdown. All are linked from `llms.txt` and advertised via `<link rel="alternate" type="text/markdown">` in the page head.

#### 4. Wikipedia Entity Auto-Linking

Build-time middleware links the first occurrence of glossary terms to the correct-language Wikipedia article. Links are baked into the static HTML — visible to JS-less AI crawlers, no client-side overhead.

#### 5. Free Audit Tools

Two serverless tools that generate leads:
- **Performance Audit** — proxies Google PageSpeed Insights server-side, returns real Lighthouse scores and Core Web Vitals
- **AI Visibility Check** — checks if a domain's `robots.txt` and `llms.txt` are accessible to AI crawlers

Both are real, functional tools — not fake lead magnets.

#### 6. GSAP Motion with Progressive Enhancement

Scroll-triggered animations (homepage disciplines, process timeline) gracefully degrade: every animated component checks `prefers-reduced-motion` and server-renders the finished state. No-JS users and crawlers see content, not blank space.

### Architecture

```
Static prerendering (Astro) → Vercel Edge
  ├── Marketing pages (DE root, EN /en)
  ├── Blog with RSS (bilingual, partner-paired)
  ├── Glossary hub + per-term pages
  ├── Programmatic city pages (DE-only)
  ├── `.md` twins for every page
  └── API endpoints (on-demand)
        ├── /api/audit → PageSpeed Insights proxy
        └── /api/ai-access → robots/llms checker
```

### Results

- Lighthouse scores consistently ≥95 on mobile
- Full bilingual coverage with native DE slugs
- Every page emits complete schema.org `@graph`
- llms.txt + 22 `.md` fact sheets published
- Free audit tools generating leads autonomously
- Zero client-side JavaScript on marketing pages

### Challenges

#### hreflang Consistency

The initial build had a trailing-slash inconsistency — `Astro.site` stringifies with a trailing slash, so naïve concatenation produced `domain.de//page`, invalidating every hreflang tag. Fixed by stripping once in the layout and enforcing `trailingSlash: 'never'` across canonical, hreflang, sitemap, and internal links.

#### AI Crawler Access

Many security plugins and CDN defaults block everything that isn't Googlebot. The `robots.txt` had to explicitly allow each AI crawler, and the actual HTTP response headers had to be verified — `X-Robots-Tag` can override `robots.txt` silently.

#### Schema Maintenance

Every schema change required pasting a built page into the Rich Results Test. A typo in JSON-LD fails silently in the browser and costs every rich result. The solution was a `JSON.parse` verification script in the build pipeline.

### Lessons Learned

SEO, GEO, and SXO are one system — not three disciplines to staff separately. A page that loads in 0.7s (SXO) is preferred by AI crawlers (GEO) and clears Google's CWV threshold (SEO). The architecture decisions (URL shape, locale structure, schema strategy) must be made before any page exists — they're one-way doors that cost 301 migrations to change later.

The GEO layer (llms.txt + .md twins + robots.txt allowlist) costs almost nothing to implement but creates a durable advantage as AI search grows. The key insight: AI crawlers are worse at JavaScript than Googlebot. Static HTML with content in the initial response is the difference between cited and invisible.
